# 기술 도서 원고 집필 가이드 v3.0 (Writing System)

본 문서는 사내 지식 엔진 구축 시리즈의 전문성과 가독성을 유지하기 위한 원고 작성 규칙을 정의합니다.

## 1. 집필 핵심 전략 (Core Strategy)

본 도서는 단순한 매뉴얼을 넘어 **'독자의 문제를 해결하는 실무 지침서'**가 되어야 합니다.

### 1.1 이론과 실습의 황금비율

- **Why**: "이론만으로는 지루하고, 실습만으로는 응용력이 없다."
- **How**: 각 장(Chapter)은 **[개념 이해] → [따라하기] → [원리 분석]** 의 3단계 구성을 준수하십시오.

### 1.2 독자 중심의 학습 곡선 (Learning Curve)

- **심화**: "실무에선 이렇게 막아야지." (예외 처리, 최적화)

## 2. 설명의 기술 (Art of Explanation)

- **원칙**: 모든 코드, 명령어, 구조도에는 **"왜 이렇게 했는지(Why)"**에 대한 설명이 반드시 포함되어야 합니다.
- **Good Patterns**:
  - "이 폴더 구조를 선택한 이유는 나중에 메타데이터로 활용하기 위함입니다."
  - "이 옵션을 켠 이유는 메모리 누수를 방지하기 위해서입니다."
- **Bad Patterns**:
  - (설명 없이 코드만 덩그러니 있는 경우)
  - "다음 명령어를 입력하세요." (이유 설명 부재)

## 3. 핵심 어조 (Tone of Voice)

- **권위 있는 하십시오체**: 전문 기술 서적의 신뢰감을 위해 정중하고 절제된 말투(~합니다, ~입니다)를 사용합니다.
- **확신 있는 단정적 서술**: "~인 것 같다", "~해 보인다" 등 추측성 표현을 금지합니다.
- **친절한 멘토링**: 독자를 함께 프로젝트를 완수하는 동료로 대우하며 실질적인 해결책을 제시합니다.

## 4. 문체 및 편집 규칙

- **능동형 중심 서술**: 주체가 명확한 능동형 문장을 사용합니다. (예: "시스템이 데이터를 수집합니다")
- **전문 용어 영문 병기**: 전문 용어는 `한국어(English)` 형식으로 표기합니다. (예: 검색 증강 생성(RAG))
- **단계적 절차 표현**: AI의 사고 과정이나 설치 순서 등 순서가 중요한 내용은 반드시 **숫자 목록(1., 2., 3.)**을 사용합니다.
- **이모지 사용 금지**: 가독성과 전문성을 위해 본문 내 이모지 사용을 금지합니다.
- **이미지 캡션 스타일**: 이미지 하단에 `*그림 X-Y: 설명*` 형식(기울임꼴 및 별표 사용)으로 캡션을 작성합니다.
- **강조 표기 규칙(Strict Spacing)**: 볼드체(`**`) 사용 시 **앞뒤 공백**을 반드시 포함하여 렌더링 오류를 방지합니다.
  - 특히 한국어와 영문 병기(`**용어(English)**`)처럼 괄호와 영문이 볼드체 내부에 포함될 경우, 마크다운 파서가 뒤에 붙는 조사와 혼동하여 렌더링이 깨지는 경우가 빈번합니다.
  - **해결책**: 볼드체가 끝나는 지점 뒤에 반드시 **한 칸의 공백**을 추가한 뒤 조사를 작성합니다.
  - 예시: `**컨텍스트 주입(Context Injection)** 이라 합니다.` (O)

### 4.1 지양해야 할 표현 (Anti-Patterns)

- **상투적 비유 금지**: "긴 여정", "마법 같은", "첫걸음" 등 감성적/문학적 표현 대신 "과정", "절차", "시작" 등 건조하고 명확한 단어 사용.
- **주관적/informal 조사 금지**: "하필", "하필이면" 등 부정적 뉘앙스나 주관적 판단이 개입된 군더더기 단어 사용 금지.
- **시의성 의존 표현 금지**: "최신", "요즘 뜨는", "현재" 등 시간 흐름에 따라 의미가 변하는 단어 대신 구체적인 버전이나 연도 표기.
- **불필요한 부연 금지**: "(=Look up)"과 같은 괄호 설명이나 족족사(Redundancy) 배제. 전문 용어는 최초 1회 정의 후 그대로 사용.

### 4.2 박스 스타일 가이드 (Admonitions)

독자의 주의를 환기시키거나 팁을 제공할 때는 아래의 표준 형식을 따릅니다.

- **팁**: 유용한 정보나 꿀팁

  > **💡 팁: 제목**
  > 내용...

- **주의**: 잠재적인 문제나 흔한 실수

  > **⚠️ 주의: 제목**
  > 내용...

- **경고**: 데이터 손실이나 치명적인 오류 위험

  > **🔥 경고: 제목**
  > 내용...

- **참고**: 부가 설명이나 참조 링크

  > **📝 참고: 제목**
  > 내용...

- **질문**: Q&A 형식 (자주 묻는 질문)
  > **❓ 질문: 질문 내용?**
  > 답변...

## 5. 실습 및 코드 작성 가이드라인 (Actionable Steps)

초보 독자가 망설임 없이 따라 할 수 있도록 **'행동(Action)'**을 구체적으로 지시합니다.

### 5.1 코드 작성 표준

- **네이밍 규칙**: 변수/함수는 `snake_case`, 클래스는 `PascalCase`를 준수합니다.
- **주석 및 힌트**: 모든 함수에는 역할과 인자를 설명하는 Docstring을 포함하며, 가능한 Type Hinting을 적용합니다.

### 5.2 실습 진행 규칙

- **Git Clone 우선**: 실습은 제공된 GitHub 저장소를 클론하여 진행하는 것을 기본으로 합니다.
- **실습 폴더 이동**: 명령어(`cd`) 대신 **"~ 폴더로 이동합니다"**와 같이 자연스러운 서술형으로 안내합니다.
- **코드 전체 제공**: 코드를 설명할 때는 생략 없이 전체 코드를 보여주어 복사/붙여넣기가 가능하게 합니다.
- **OS별 명령어 분리**: 설치 명령어 등 OS에 따라 달라지는 경우 **macOS**와 **Windows**를 구분하여 제공합니다.
- **코드 로직 박스**: 각 실습 단계마다 `#### 🔍 코드는 어떻게 작동하나요?` 섹션을 추가하여 코드의 핵심 기각 원리(알고리즘, 정규표현식 등)를 설명합니다.
- **이중 언어 디지털화**: 이미지 분석(LLaVA) 등 멀티모달 실습 시 한국어와 영문 설명을 동시에 생성하도록 유도하여 검색 확장성을 확보합니다.
- **2단계 데이터 파이프라인**: PDF 지능형 정제 시 반드시 **'PDF -> Text 추출 -> AI 정제'** 라는 2단계 파이프라인을 강조하고, AI가 텍스트 기반으로 동작하는 이유를 설명합니다.

### 5.3 코드 설명 표준 (Code Explanation Standard)

- **헤더 규칙**: 코드 블록 바로 아래에는 반드시 **`#### 🔍 코드 워크플로우 (Code Workflow)`** 섹션을 배치합니다.
- **내용 구성**: 코드가 실행되는 논리적 순서(Step-by-Step)를 번호 매기기 목록(`1.`, `2.`)으로 설명하십시오.
  - **입력(Input)**: 함수나 클래스로 들어오는 데이터
  - **처리(Process)**: 핵심 로직 1~2줄 요약
  - **출력(Output)**: 반환값 또는 생성되는 파일

### 5.4 [공통] 실습 환경 구성 원칙

본 시리즈의 모든 실습은 독자가 쉽게 따라 할 수 있는 환경을 제공해야 합니다.

1.  **의존성 최소화**: 복잡한 설치 과정 없이 `pip install` 한 번으로 시작할 수 있게 `requirements.txt`를 제공합니다.
2.  **데이터 포함**: 실습에 필요한 데이터(PDF, 이미지 등)는 저장소에 미리 포함시켜 독자가 별도로 구할 필요가 없게 합니다.
3.  **결과물 검증**: 각 단계마다 '예상 실행 결과'를 텍스트나 스크린샷으로 제시하여 독자가 올바르게 따라오고 있는지 확인시켜줍니다.

## 6. 시각화 및 도구(Tool) 가이드

- **Mermaid 간소화**: 하나의 다이어그램에 노드가 7개를 넘지 않도록 단순화하여 가독성을 확보합니다.
- **도구(Tool) 소개**: 새로운 도구 등장 시 아래 포맷을 사용합니다.
  - **도구명**: `[Tool Name]`
  - **기능**: 무엇을 하는 도구인가?
  - **입력 스키마**: 어떤 데이터를 받는가? (JSON 예시)

## 7. 원고 구조 템플릿

1. **헤더 및 요약**: 단원의 목적을 1~2문장으로 요약.
2. **개념 설명**: Mermaid 도식을 활용하여 아키텍처나 원리를 먼저 설명.
3. **단계별 실습**: 명확한 코드 블록과 라인별 주석 포함.
4. **마무리 및 연결**: 학습 내용 요약 및 다음 장과의 연계성 제시.

### 7.1 챕터 마무리 규칙 (Chapter Conclusion Rule)

- **섹션 제목**: 각 장의 마지막 섹션 제목은 반드시 **"정리하며"** 로 통일합니다. (예: `## 3. 정리하며`)
- **내용 구성**:
  - **개조식(Bullet Points)**: 줄글로 나열하지 말고, 핵심 내용을 불렛 포인트로 정리하십시오.
  - **두괄식(Deductive)**: 각 항목의 첫 문장에 핵심 결론을 먼저 제시하고, 부연 설명은 뒤에 덧붙이십시오.

## 8. 실전 체크리스트 (Quality Control)

집필 완료 전 반드시 아래 항목을 체크하십시오.

- [ ] **기술 명칭**: 기술 용어(Technical Terms)가 대문자로 정확히 표기되었는가?
- [ ] **볼드체 공백**: 기술 용어 강조 시 앞뒤 공백(`**Term**`)이 포함되었는가?
- [ ] **실습 동기화**: 본문의 코드가 `hands-on` 폴더의 코드와 일치하는가?
- [ ] **이미지 경로**: 실제 이미지 경로가 올바르게 입력되었는가? (`./images/...`)
