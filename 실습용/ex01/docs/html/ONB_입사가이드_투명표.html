3. 커밋 메시지 규칙 (Commit Convention)

💡 작성 Tip 커밋 메시지는 동료를 위한 배려입니다.
'제목'은 50자 이내로 요약하고, '본문'에는 변경 이유를 상세히 적어주세요.

우리 팀은 Git 커밋 메시지의 가독성을 위해 Conventional Commits 규칙을 따릅니다. 모든 커밋 메시지는 아래 정의된 [태그]로 시작해야 하며, 파이프라인에서 자동 검사됩니다.

아래 표는 테두리가 없으므로, AI 파서가 텍스트의 정렬(Alignment)만 보고 행과 열을 구분해야 합니다.

태그 (Tag) 설명 (Description) 사용 예시 (Example)
Feat 새로운 기능을 추가할 때 사용합니다. (예: API 개발) Feat: 사용자 프로필 조회 API 추가
Fix 버그를 수정할 때 사용합니다. (긴급 수정 포함) Fix: 로그인 시 발생하는 401 에러 해결
Refactor 코드 리팩토링 (기능 변경 없이 구조 개선) Refactor: 인증 미들웨어 로직 분리
Style 코드 포맷팅, 세미콜론 누락 등 UI/스타일 수정 Style: 버튼 컴포넌트 마진값 조정
Docs 문서를 수정하거나 추가할 때 사용합니다. Docs: API 엔드포인트 명세서 업데이트
Test 테스트 코드를 작성하거나 수정할 때 사용합니다. Test: 결제 흐름 통합 테스트 코드 작성
Chore 빌드 설정이나 패키지 매니저 등 단순 관리 작업 Chore: 프로젝트 의존성 라이브러리 업데이트
