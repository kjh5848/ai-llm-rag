# [Strategy 07] RAG 아키텍처 전략: Rule-based vs AI-based

본 문서는 PDF 파싱 과정에서 발생하는 **"코드로 어디까지 짜야 하고, AI에게 어디부터 맡겨야 하는가?"** 라는 핵심 딜레마에 대한 아키텍처 가이드입니다.

---

## 1. 핵심 딜레마 (The Core Dilemma)

PDF 파싱에는 두 가지 극단적인 접근 방식이 존재합니다.

| 구분 | Rule-based (Python/Pandas) | AI-based (LLM/VLM) |
| :--- | :--- | :--- |
| **방식** | 사람이 `if-else` 로직과 좌표(`bbox`)를 직접 코딩. | 문서를 통째로 AI에게 던져서 마크다운 변환 요청. |
| **장점** | **정확성(Precision)**. 숫자가 틀리지 않음. 무료. 빠름. | **범용성(Universal)**. 레이아웃이 바뀌어도 알아서 이해함. |
| **단점** | **취약성(Fragile)**. 레이아웃이 1mm만 밀려도 에러 발생. | **비용(Cost)** & **환각(Hallucination)**. 숫자를 틀릴 수 있음. |

> **Q. 모든 케이스를 if문으로 짜야 하나요?**  
> **A. 아니오. 불가능합니다.** 세상의 모든 PDF 레이아웃을 코드로 대응할 수는 없습니다.  
> 
> **Q. 그럼 전부 LLM에게 맡기면 되나요?**  
> **A. 위험합니다.** 재무제표의 숫자 하나만 틀려도 RAG의 신뢰도가 바닥으로 떨어집니다.

---

## 2. 해결책: 3단계 진화 (The 3-Stage Evolution)

현업에서는 다음과 같이 단계적으로 시스템을 고도화합니다.

### ✅ 1단계: 휴리스틱 분석 (Heuristic) - *현재 실습 단계*
*   **도구**: `pdfplumber`, `PyMuPDF`
*   **로직**: "선이 있으면 표다", "좌표가 겹치면 병합한다".
*   **용도**: 정형화된 보고서, 견적서, 공고문 등 레이아웃이 고정된 경우.
*   **한계**: 테두리 없는 표, 복잡한 다단 문서 취약.

### ✅ 2단계: 레이아웃 분석 모델 (Layout Analysis Model)
*   **도구**: Microsoft Azure DI, Naver Clova OCR, LayoutLM
*   **로직**: AI 모델이 문서의 **구조(표, 그림, 제목, 본문)**를 먼저 식별하여 좌표를 줍니다.
*   **변화**: 개발자가 일일이 `find_tables()` 로직을 짤 필요가 없습니다. "표 영역 좌표"를 받아서 그 부분만 텍스트로 추출하면 됩니다.

### ✅ 3단계: VLM (Vision Language Model) - *최종 병기*
*   **도구**: GPT-4o, Gemini 1.5 Pro
*   **로직**: PDF를 **이미지**로 변환하여 AI에게 시각적 이해를 요청.
*   **용도**: 난해한 도표, 수기 작성 문서, 스캔본, 복잡한 인포그래픽.

---

## 3. 추천 아키텍처: 하이브리드 라우터 (Hybrid Router)

가장 효율적인 방법은 문서를 분류하여 처리 방식을 다르게 가져가는 **라우터(Router)** 시스템입니다.

```mermaid
graph TD
    A[PDF 입력] --> B{문서 분류기 (Router)}
    
    B -- "Type 1: 정형 데이터" --> C[Rule-based Parser]
    B -- "Type 2: 텍스트 위주" --> D[Text Extractor]
    B -- "Type 3: 비정형/복잡" --> E[VLM / Layout Model]
    
    C[Python/Pandas] --> F[Markdown 통합]
    D[PyMuPDF] --> F
    E[Gemini/GPT-4o] --> F
    
    F --> G[Vector DB 저장]
```

### 🛠️ 전략적 선택 기준
1.  **숫자/데이터의 정확성**이 생명이라면? 👉 **Rule-based (Python)**
    *   예: 재무제표, 성적표, 실험 데이터
2.  **전체적인 맥락(Context)** 이해가 중요하다면? 👉 **Text Extractor**
    *   예: 계약서, 논문, 뉴스 기사
3.  **구조가 복잡하고 매번 바뀐다면**? 👉 **VLM (AI)**
    *   예: 카탈로그, 홍보물, 스캔된 옛날 문서

---

## 4. 결론 (Conclusion)

지금 우리가 실습하는 `pdfplumber` + `pandas` 로직은 **Type 1 (정형 데이터)** 처리를 위한 가장 강력하고 경제적인 기초 체력입니다.

이 기초 위에, 해결되지 않는 예외 케이스(Case 3, 4 등)를 만났을 때만 **VLM**을 호출하는 형태로 시스템을 확장하는 것이 **비용 효율적인 RAG 파이프라인**의 핵심입니다.
