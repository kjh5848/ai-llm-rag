# PDF RAG 데이터 처리 실습 (PDF RAG Handling)

이 프로젝트는 PDF 문서를 RAG(Retrieval-Augmented Generation) 시스템에 통합할 때 발생하는 **다양한 실패 케이스**와 그 **해결 전략**을 실습하는 공간입니다.

## 📂 실습 케이스 목록 (Overview)

| ID | 케이스명 (Case Name) | 난이도 | 상태 | 핵심 문제 (RAG Failure) | 해결 전략 (Key Strategy) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **01** | 다단 편집 (Multi-Column) | Easy | ✅ 완료 | 텍스트가 좌우로 섞임(Shredding) | `layout=True`, `x_tolerance` 조절 |
| **02** | 표 데이터 (Tables) | Medium | ✅ 완료 | 투명표/대형표 구조 파괴 | 그리드 재구성(X-Y Bucketing) |
| **03** | 노이즈 제거 (Noise/Header) | Easy | ✅ 완료 | 페이지 번호, 헤더가 본문에 섞임 | `crop(bbox)` 영역 지정 추출 |

| **05** | 시각 자료 (Vision/Charts) | Hard | ✅ 완료 | 차트/그래프의 의미 파악 불가 | VLM (Vision Language Model) 설명 생성 |

## 🔍 상세 가이드 (Case Details)

### 1단계: 텍스트 및 레이아웃 기초 (Fundamentals)

#### **01. 다단 편집 (Multi-Column)**
*   **문제**: 논문이나 뉴스레터처럼 2단/3단으로 된 문서를 그냥 읽으면, 왼쪽 단의 줄과 오른쪽 단의 줄이 합쳐져서 "외계어"가 됩니다.
*   **해결**: `pdfplumber`의 `layout=True` 옵션이나 `vertical_strategy='text'`를 사용하여 물리적 레이아웃을 인식시킵니다.

#### **02. 표 데이터 (Tables)**
*   **문제**: PDF의 표는 시각적인 선형일 뿐, 실제로는 텍스트 덩어리입니다. 이를 줄글로 읽으면 행/열 관계가 모두 파괴됩니다.
*   **해결**: `extract_table()` 메서드로 격자 구조를 복원하고, 이를 CSV나 Markdown Table 형태로 변환하여 LLM에 제공합니다.

#### **03. 노이즈 제거 (Noise/Header & Footer)**
*   **문제**: 매 페이지마다 반복되는 "회사명", "날짜", "페이지 번호"가 검색 품질을 저하시킵니다 (Top-K 오염).
*   **해결**: 페이지 상/하단의 일정 영역(예: 상위 10%, 하위 10%)을 `crop()`으로 잘라내고 본문만 추출합니다.

---

### 2단계: 이미지 및 복합 문서 (Advanced Content)

#### **04. OCR & 비전 (OCR & Vision)**
*   **문제**: 디지털 텍스트가 없는 "이미지 PDF"이거나, 손글씨/구겨진 문서처럼 일반적인 추출이 불가능한 경우입니다.
*   **해결**: 기초적인 텍스트 추출은 로컬 엔진인 **EasyOCR**을 사용하고, 복잡한 레이아웃 복원이나 손글씨/차트 해석이 필요할 때는 **VLM(Vision Language Model)**을 연동하여 마크다운으로 변환합니다.

#### **05. 시각 자료 (Vision/Charts)**
*   **문제**: "매출 추이 그래프"는 텍스트로 표현되지 않습니다. RAG가 "매출이 올랐나요?"라는 질문에 답할 수 없습니다.
*   **해결**: 차트 영역을 이미지로 캡처하여 Vision LLM(GPT-4o 등)에게 "이 차트를 해석해 줘"라고 요청하고, 그 설명을 텍스트로 저장합니다.

#### **06. 복합 레이아웃 (Complex)**
*   **문제**: 교과서나 잡지처럼 그림 주위에 글자가 흐르는 레이아웃입니다.
*   **해결**: 레이아웃 분석 모델(LayoutLM 등)이나 좌표 기반 규칙을 사용하여 "그림", "표", "본문" 영역을 먼저 나누고 각각 다르게 처리합니다.

---

### 3단계: 실무 엣지 케이스 (Edge Cases)

#### **07. 회전된 페이지 (Rotation)**
> *"표가 너무 커서 가로로 눕혀놨어요."*
*   **RAG 실패 원인**: 텍스트 추출기는 기본적으로 문서를 "세로"로 읽습니다. 가로로 눕혀진 텍스트는 깨지거나 순서가 뒤집힙니다.
*   **해결책**: 페이지의 `rotation` 속성을 확인하여 90도/270도 회전된 경우, 이미지를 바로잡거나 좌표축을 변환하여 읽습니다.

#### **08. 양식 필드 (Form Fields)**
> *"신청서의 체크박스(☑) 값을 못 읽어요."*
*   **RAG 실패 원인**: PDF의 Interactive Form(AcroForm) 데이터는 일반 텍스트 레이어와 다른 방식으로 저장됩니다.
*   **해결책**: PDF의 `Annots`(주석/폼) 딕셔너리를 조회하여 위젯(Widget)의 상태 값(`/V`, `/AS`)을 추출합니다.

#### **09. 주석 및 마스킹 (Annotations)**
> *"중요하다고 형광펜 칠했더니 글자가 안 읽혀요."*
*   **RAG 실패 원인**: 하이라이트나 메모(Sticky Note) 객체가 텍스트 위에 덮어씌워져 있어, 추출기가 이를 "가려진 텍스트"로 판단하거나 주석 내용만 읽어버립니다.
*   **해결책**: 텍스트 추출 시 `annotations` 객체를 제외하거나, 좌표상 겹치는 영역을 우선순위 처리합니다.

#### **10. 벡터 그래픽 (Vector Paths)**
> *"디자인된 제목인데 드래그가 안 되고 OCR도 안 먹혀요."*
*   **RAG 실패 원인**: 글꼴(Font)을 사용하지 않고, 일러스트레이터 등에서 글자를 '곡선(Path)'으로 깨서 저장한 경우입니다. 컴퓨터 입장에서는 그냥 그림입니다.
*   **해결책**: 텍스트 추출이 불가능하므로, 해당 영역을 비트맵 이미지로 래스터화(Rasterize)한 후 강제로 OCR을 수행해야 합니다.
