# 📖 Case 04: OCR 문자인식 (EasyOCR vs VLM)

## 1. 개요 (Summary)
- **난이도 (Difficulty)**: 상 (High)
- **주요 기술 (Key Tech)**: EasyOCR (Local), GPT-4o Vision (Cloud)
- **핵심 개념 (Core Concept)**: "눈에 보인다고 모두 같은 데이터는 아닙니다." 문서의 상태(화질, 오염도, 필기체)에 따라 **비용 효율성**과 **복구 능력** 사이의 최적 도구를 선택해야 합니다.

## 2. 케이스 분석 (Case Analysis)

### 🔹 Case 1: 표준 인보이스 (EasyOCR)
- **입력 (Input)**: `04_case1_EasyOCR.pdf`
- **문제 (Problem)**: **"정형 데이터의 대량 처리"**. 깨끗한 영수증이나 청구서는 굳이 비싼 LLM을 쓸 필요가 없습니다.
- **해결 (Solution)**: **Local OCR**. EasyOCR이나 Tesseract 같은 로컬 엔진으로 비용 "0"으로 텍스트를 추출합니다.

### 🔹 Case 2: 오염된 스캔 계약서 (VLM)
- **입력 (Input)**: `04_case2_스캔문서_VLM.pdf`
- **문제 (Problem)**: **"물리적 손상"**. 구겨지거나 얼룩이 묻어 기존 OCR이 글자를 "노이즈"로 인식합니다.
- **해결 (Solution)**: **VLM(Vision Language Model)**. 이미지를 픽셀 단위로 읽지 않고, 전체적인 "의미"를 추론하여 글자를 복구합니다.

### 🔹 Case 3: 손글씨 (Handwriting)
- **입력 (Input)**: `04_case3_손글씨.pdf`
- **문제 (Problem)**: **"비정형 필기체"**. 사람마다 제각각인 글씨체를 일반 OCR은 거의 인식하지 못합니다.
- **해결 (Solution)**: **LLM Context Inference**. "처방전", "메모"라는 문맥을 주면, VLM이 흘려 쓴 글씨도 문맥상 알맞은 단어로 보정하여 읽어냅니다.

### 🔹 Case 4: 저화질 (Low Resolution)
- **입력 (Input)**: `04_case4_저화질.pdf`
- **문제 (Problem)**: **"픽셀 뭉개짐"**. 팩스로 받은 문서처럼 글자 경계가 흐릿하여 OCR이 오타를 남발합니다.
- **해결 (Solution)**: **Generative Correction**. "이 문서는 ㅇㅇ에 관한 내용이야"라고 힌트를 주면, 생성형 AI가 흐릿한 글자를 문맥에 맞게 "다시 써서(Re-generation)" 반환합니다.

## 3. RAG 구축 전략 (RAG Strategy)

OCR 및 비전 처리의 핵심은 **"비용 효율적인 정확도 확보"** 입니다. 모든 페이지를 비싼 VLM으로 처리하는 것이 아니라, 문서의 상태에 따라 로컬 엔진과 클라우드 AI를 적절히 배분하여 RAG 시스템의 운영 지속성을 확보해야 합니다.

| 문제 상황 | RAG에 미치는 영향 (Risk) | 해결 전략 (Solution) |
| :--- | :--- | :--- |
| **정형화된 인보이스 (Case 1)** | 단순 반복적인 텍스트 추출에 과도한 AI 비용이 발생하여 토큰 효율성이 저하됩니다. | **로컬 OCR (EasyOCR)**: 비용이 들지 않는 로컬 엔진으로 텍스트를 빠르게 추출하여 처리량을 극대화합니다. |
| **오염된 스캔본 (Case 2)** | 구겨지거나 얼룩진 부분의 글자가 깨어짐으로써 검색 키워드 매칭이 불가능해집니다. | **VLM 추론 (GPT-4o)**: 픽셀 인식을 넘어 문맥적 의미를 파악하는 비전 인공지능으로 단어를 복원합니다. |
| **손글씨 및 저화질 (Case 3, 4)** | 흘려 쓴 글씨나 뭉개진 픽셀로 인해 검색 엔진에 'Garbage 데이터'가 쌓여 답변 신뢰도가 하락합니다. | **문맥 기반 보정 (Context Inference)**: 전후 문맥 정보를 AI에게 힌트로 주어 오타를 생성적으로 교정하여 추출합니다. |

> **본 실습의 핵심**:
> - **하이브리드 파이프라인**: 깨끗한 페이지는 EasyOCR로, 읽기 힘든 페이지는 VLM으로 처리하는 조건부 분기 로직을 구축하는 것이 실무 RAG 전처리의 핵심 노하우입니다.

## 4. RAG 비교 분석 (Success vs Fail)

| 구분 | 파일 | 포맷 (Format) | RAG 처리 결과 (예시) |
| :--- | :--- | :--- | :--- |
| **성공 (Success)** | `04_caseX_..._성공.md` | Contextual Text | `(손글씨) 환자에게 아스피린 처방` <br> -> **"정확한 약물명 검색 가능"** |
