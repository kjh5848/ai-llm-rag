# 6장. VectorDB 구축: 파편화된 문서를 지식의 결정체로

5장에서 우리는 '비정형적이고 복잡한' 사내 문서들을 AI 친화적인 마크다운(Markdown)으로 정제하는 전략을 세웠습니다. 본 장에서는 그 전략을 코드로 구현하여, 파편화된 파일들을 **숫자(벡터)** 로 변환하고 AI가 초고속으로 검색할 수 있는 **지식 창고(VectorDB)** 에 적재하는 전체 파이프라인을 직접 실습해 보겠습니다.

---

## 1. 전처리의 역설: 왜 문서 전략(Document Strategy)이 필요한가

본격적인 코드 실습에 앞서 한 가지 근본적인 질문을 던져보겠습니다. **"왜 우리는 이렇게 복잡한 예외 처리 코드를 짜야만 할까요?"**

정답은 **서식(Format) 중심의 인간친화적 문서 작성 관행** 때문입니다.
단순히 보기 좋게 만들기 위해 다단을 나누고, 셀을 불규칙하게 병합하며, 폰트 크기로 무작정 대소제목을 구분하는 방식은 필연적으로 구조 파괴를 낳습니다. 결국 이런 엉망진창인 '비정형 데이터'를 구출하기 위해 우리는 `pdfplumber`, `openpyxl`, 심지어 거대 모델인 Vision LLM까지 동원하는 무거운 전처리 파이프라인(Preprocessing Pipeline)을 가동해야만 하는 역설에 빠지게 됩니다.

반대로 생각해 보겠습니다. 만약 사내 문서 작성 가이드(Document Strategy)가 철저히 지켜진다면 어떻게 될까요?
1. 엑셀의 헤더(컬럼명)를 병합 없이 1줄로 유지한다.
2. 워드는 시각적 폰트 크기가 아닌 '스타일 속성(제목 1, 2 등)'을 엄격히 적용해 작성한다.

이러한 **'문서의 표본화 및 규칙화'가 선행된다면 전처리 로직은 믿을 수 없을 만큼 단순화**됩니다. 단순한 마크다운 추출기 하나만으로 완벽한 품질의 텍스트를 얻어낼 수 있으며, 파싱 실패(Fail)율은 0%에 수렴하게 됩니다. 즉, **가장 훌륭한 파싱 코드는 올바른 문서 기획(Strategy) 그 자체입니다.**

하지만 현실의 기업 환경에서는 과거의 유산(Legacy docs)과 통제 불가능한 파일들이 차고 넘칩니다. 따라서 우리는 최악의 상황을 가정하고, 어떠한 악조건의 파일이 들어오더라도 이를 정제해 내는 고도화된 파이프라인을 구축할 것입니다.

## 2. 지식 저장소 구축 로드맵

우리가 구축할 RAG 시스템의 뼈대는 다음과 같습니다. 이 흐름을 머릿속에 담아두십시오.

```mermaid
graph LR
    A["난해한 원본 문서<br/>(PDF, Excel, Word)"] -- "1~4번 스크립트" --> B["표준 마크다운<br/>(.md)"]
    B -- "5번 스크립트<br/>(Chunking & Embedding)" --> C["지식 저장소<br/>(ChromaDB)"]
    C -- "6번 스크립트<br/>(Similarity Search)" --> D{"사용자 질문에<br/>정답 반환"}
```

1. **파싱 (Parsing)**: 제각각인 문서의 껍데기를 벗겨내고 순수한 마크다운 지식으로 정제합니다.
2. **청킹 및 적재 (Ingestion)**: 텍스트를 AI 모델이 소화하기 좋은 크기로 자르고, 벡터 수학 공간에 꽂아 넣습니다.
3. **유사도 검색 (Search)**: 사용자의 질문이 들어왔을 때 가장 유사한 맥락을 즉시 찾아냅니다.

## 4. [실습] 파싱 파이프라인 가동 (A/B 테스트)

이제 각 문서의 특성에 맞춰 단순 텍스트 추출(Fail)과 맞춤형 파싱(Success) 결과가 어떻게 다른지 **A/B 테스트** 형식으로 실행해 보겠습니다.

### 실습 1. 다단 PDF 구조화 파싱

HR 부서의 취업규칙은 2단으로 나뉘어 있고 중간에 표가 섞여 있습니다. 먼저 단순 파싱이 어떻게 실패하는지 확인한 후, `pdfplumber`를 활용하여 레이아웃을 보존하며 추출해 보겠습니다.

```bash
# 1-1. 단순 파싱 (실패 사례)
python 01_pdf_parsing/parse_fail.py

# 1-2. 고도화 파싱 (성공 사례)
python 01_pdf_parsing/parse_success.py
```

* **실행 결과 비교**: `parsed_data/fail_results`와 `success_results` 폴더를 확인해 보십시오.
* 단순 파싱본은 왼쪽 단과 오른쪽 단의 글이 섞이는(Shredding) 현상이 발생합니다. 반면 고도화 파싱본은 문맥이 보존되고, 내부 징계 표가 완벽한 마크다운 테이블(`| --- |`)로 변환됩니다.

## 5. [실습] 지식 적재: 텍스트를 숫자로 (5번 스크립트)

이제 `parsed_data/` 폴더에 완벽한 마크다운 지식들이 모였습니다. 이 텍스트들을 AI가 계산할 수 있는 **'다차원 벡터(숫자 모음)'** 로 변환하여 저장할 차례입니다.

```bash
python 5_ingest_to_vector.py
```

### 코드 워크플로우 (Code Workflow)

1. **마크다운 청킹 (MarkdownHeaderTextSplitter)**: 
   긴 문서를 무식하게 자르지 않습니다. `##`, `###` 같은 헤더 기준으로 1차 분할하여 **'문맥 단위'**를 유지합니다.
2. **임베딩 (HuggingFaceEmbeddings)**: 
   "비밀번호"와 "패스워드"는 모양이 다르지만 벡터 공간에서는 비슷한 위치(숫자)를 갖게 됩니다. 우리는 실습을 위해 가볍고 한국어에 강한 `bge-m3` 모델을 사용합니다.
3. **적재 (ChromaDB)**: 
   변환된 벡터 좌표와 원문 조각을 영구적인 데이터베이스(`chroma_db/`)에 밀어 넣습니다.

> **💡 팁: 왜 클라우드가 아닌 로컬 Chroma DB인가요?**
> 사내 데이터는 절대 외부 API(OpenAI 등)로 새어나가선 안 되는 보안 등급을 갖는 경우가 많습니다. 로컬 ChromaDB는 네트워크 지연(Latency)이 없고, 비용이 100% 무료이며, 완벽한 보안을 자랑합니다.

## 7. 정리하며

여러분은 방치되어 썩어가던 쓰레기 문서(Garbage)들을 **AI가 언제든 빼 쓸 수 있는 정제된 지식 탄약고**로 탈바꿈시켰습니다.

* **맞춤형 파싱**: 무식한 텍스트 추출이 아닌, 엑셀, PDF 등 구조의 파괴를 막는 파싱 전략을 구현했습니다.
* **마크다운 청킹(Chunking)**: 단순 글자 수 쪼개기가 아니라 마크다운 헤더(`#`) 기준으로 논리적 조각을 내었습니다.
* **임베딩과 벡터 배열**: 추출된 지식을 다차원 벡터 공간에 배치하고 유사도 검색 환경을 완성했습니다.

이제 지식은 준비되었습니다. 다음 **7장**에서는 이 저장소에 파이프를 연결하여, 질문에 대해 유려하게 대답하는 **대화형 RAG 챗봇**의 두뇌를 조립해 보겠습니다.
